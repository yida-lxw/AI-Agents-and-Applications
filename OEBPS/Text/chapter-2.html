chat.completions.create 函数

chat.completions.create函数是 OpenAI Completions API 的一部分，是与 OpenAI 大语言模型交互的主要方式。理解其函数签名至关重要：

client.chat.completions.create(
  model="gpt-5-nano",
  messages=[
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": prompt_input}
  ] 
)

以下是参数的解释说明：

  • model—它指的是您希望使用的OpenAI模型。可选范围从GPT-5（最准确但也最慢且最昂贵）到GPT-5-nano（最快且最便宜，尽管在某些任务上准确性较低）。GPT-5-mini提供了一个稳健的中间选择，在速度、精确度和成本之间取得了平衡。
  • messages—提示词被表达为一个消息列表，每条消息都有一个角色（例如，系统、用户或助手）和一些包含需要遵循的指令或文本的内容。这种格式遵循了OpenAI构建对话输入的规范。

尽管Responses API现已成为OpenAI的首选接口—尤其适用于基于智能体的工作流—但本书主要使用Completions API，通常通过LangChain中的ChatOpenAI包装器间接访问。原因是Completions不仅受到OpenAI模型的支持，还受到许多开源大语言模型的支持，为您提供了更大的灵活性。我们将在第11章深入探讨AI智能体时，重新讨论Responses API。

2.3 提示词模板

在构建大语言模型应用时，创建能够通过参数整合用户输入的灵活提示词模板至关重要。LangChain通过其PromptTemplate类简化了这一过程，它允许您管理和复用参数化提示词，而无需编写自定义函数。这不仅简化了动态提示词的创建，还提升了大语言模型交互的效率和适应性，稍后我将进行演示。

2.3.1 使用Python函数实现提示词模板

为了说明模板的概念，让我们创建一个文本摘要模板，它需要请求文本内容、期望的摘要长度以及偏好的语气风格。你可以通过一个简单的Python函数来实现：

def generate_text_summary_prompt(text, num_words, tone):
    return f"You are an experienced copywriter. Write a 
    ↪{num_words} words summary of the following text, using a 
    ↪{tone} tone: {text}"

让我们使用提示词模板来生成一个提示词，然后像往常一样通过LangChain的ChatOpenAI包装器来执行它：

2.4.1 文本分类

在分类任务中，目标是将输入文本分配到多个预定义类别中的其中一个。以下提示词展示了这一概念—请尝试将其输入到ChatGPT中，我们将在本节中全程使用它：

2.4.6 推理

虽然推理是一项强大的能力，但大语言模型在这方面仍在不断改进。因此，你可能需要更高级的提示词工程技术来获得想要的结果。以下是一个涉及数学推理的简单示例：

 指令 计算这串数字中平方数的和：19, 13, 1, 17, 4, 64, 900

“这串数字19, 13, 1, 17, 4, 64, 900中是平方数的有：1, 4, 64, 900，它们的和为：”

1 + 4 + 64 + 900 = 969

因此，这串数字中平方数的和为969。

ChatGPT识别出平方数，将它们挑选出来，然后计算它们的总和。下一个例子更具挑战性，涉及到回文数—那些正读反读都相同的数字，例如1661或27872：

 指令 求这串数字中回文数的和：13, 1331, 121, 73, 99, 56, 232, 7

我得到了如下输出：

“这串数字13, 1331, 121, 73, 99, 56, 232, 7中是回文数的有：1331, 121, 232, 7，它们的和为：”

因此，这串数字中回文数的和为1691。

这是错误的！回文数是指正读和反读都相同的数字。因此，99本应被视为回文数，但它却没有被纳入计算。

2.5 详细推理

在开发大语言模型应用时，特别是那些需要复杂推理的应用，理解不同提示词技术的局限性至关重要。例如，零样本学习是一种不提供任何示例的方法，期望大语言模型仅根据给定的指令来理解和执行任务。虽然这种方法在某些场景下效果良好，但对于更复杂的任务（例如教大语言模型识别回文），它并不能总产生准确的结果。在接下来的章节中，我将演示如何超越零样本学习，优化大语言模型提示词，以在推理任务中获得更可靠的性能。

2.5.1 单样本学习

当你在提示词中只提供一个示例时，这被称为单样本学习。以下是我通过提供单个示例来尝试解决回文问题的方法：

 指令 求这串数字中回文数的和：13, 1331, 121, 73, 99, 56, 232, 7

 示例 33是一个回文数

“我为之前的疏忽致歉。让我们重新正确识别回文数并计算这串数字中的回文数之和：13, 1331, 121, 73, 99, 56, 232, 7。”

“这串数字中是回文数的有：1331, 121, 232, 7，它们的和为：”

1331 + 121 + 232 + 7 = 1691

因此，这串数字中回文数的和确实为1691。

遗憾的是，即使使用了单样本学习，问题依然存在。因此我调整了提示词并加入了两个示例。

2.5.4 少样本学习

在某些情况下，仅提供两个示例可能不足以教会大语言模型理解一个新概念。此时，你需要提供多个示例，这被称为少样本学习。为了演示这一点，让我来设计一个游戏，在这个游戏中，一个数字可以被归类为Abra、Kadabra或Abra Kadabra。让我们看看在没有提供任何分类规则信息时会发生什么。

2.5.6 思维链

最后，我想向您介绍思维链技术，它结合了提供逻辑步骤和少量示例学习。在这里，我虚构了一个新的数学定义：如果一个数列包含至少两个奇数，并且所有奇数的和能被3整除，则该数列被称为“奇异”数列。首先，我检查大语言模型是否不熟悉奇异数列的规则：

“您提供的这个数列看起来是递增的，但它并不遵循常见的数学规律，比如等差数列或等比数列。它是否被视为“奇异数列”取决于您用来定义“奇异数列”的上下文或标准。从数学意义上讲，它可能被认为是一个不寻常或非标准的数列。如果您有定义“奇异”数列的具体标准或规则，请提供更多细节以便进行更精确的分类。”

正如预想的那样，ChatGPT似乎并不能理解“奇异”数列的概念，这再次证明，值得庆幸的是，它尚未发展出读心术，至少目前如此。

我将基于Wei等人提出的思维链技术（参见https://arxiv.org/abs/2201.11903），构建一个提示词模板。此提示词是对少量样本学习提示词的扩展，将包含以下内容：

在本节中，您已经了解了如何通过提供各种示例或概述必要步骤，向ChatGPT介绍一项新任务。这种方法被称为上下文学习，因为您是在利用提示词的上下文中所提供的信息来指导大语言模型。

定义  上下文学习 指的是任何通过提示词上下文中的示例，指导大语言模型完成用户请求的陌生任务或指令的方法。常见的技术包括单样本学习、双样本学习或少样本学习，以及提供逐步指导（通常称为思维链）。与微调相比—微调涉及通过在特定领域的文本上进行训练，基于通用模型创建专用的大语言模型—上下文学习是一种更具成本效益且资源密集度更低的方法。它不需要高端硬件（如GPU），也不需要深入理解Transformer架构。

思维链 (CoT) 通过将问题分解为更小的逻辑步骤，改进了大语言模型处理复杂推理的方式。然而，它存在一些局限性，例如缺乏更深入的探索或在处理混乱上下文时会遇到困难。两种先进技术—思维树（ToT）和思维线（ThoT）—可以解决这些不足：

  • 思维树(ToT)—ToT通过允许大语言模型探索多种推理路径来提升问题解决能力。传统模型进行逐个token的决策，这限制了它们提前规划或重新审视先前选择的能力。ToT将这一过程结构化为连贯的步骤，使模型能够在推进过程中评估和完善选项（参见https://arxiv.org/abs/2305.10601）。这种方法在需要战略性思考的任务中非常有效，例如24点游戏、创意写作和迷你填字游戏。例如，ToT帮助GPT-4解决了74%的24点游戏问题，而使用CoT仅解决了4%。这种方法有助于提升审慎决策、回溯和战略预见能力。
  • 思维线索(ThoT)—ThoT是一种针对混乱上下文（参见 https://arxiv.org/abs/2311.08734）中挑战的解决方案。在混乱上下文中，无关细节会干扰模型或导致错误。受人类认知过程启发，ThoT 通过系统性地分解和分析大段信息，同时聚焦相关细节，有效应对这一问题。该策略可作为模块化插件，与多种模型及提示词技术无缝集成。在 PopQA、EntityQ 以及多轮对话响应等数据集上的实验表明，ThoT 能显著提升推理准确性，其优势在于有效过滤噪声并在复杂上下文中保持注意力集中。

这两种技术都推动着大语言模型超越了简单的响应生成，在结构化和混乱的场景中提升了推理、规划和决策能力。有关技术细节和示例，请参阅已发表的研究论文和代码仓库。

2.6 提示词结构

结合前面各章节的所有提示词要素，可得出以下通用结构：

  • 角色设定—明确指定你希望大语言模型（LLM）扮演的角色。无论是经验丰富的市场营销经理、获奖的科幻作家，还是专业计算机杂志的文稿编辑，都应设定高标准的期望。
  • 上下文—提供详细的上下文信息，帮助大语言模型理解你请求所处的上下文环境。精确的描述有助于提高结果与期望的契合度。
  • 指令—清晰定义你希望大语言模型对输入文本执行的具体操作。无论是翻译、写作、改写、优化、分类、排序还是查找，都需明确说明。
  • 输入—指的是具体的上下文细节，可以是一段文字、一个问题或一系列关键点。你可以选择相应的标签进行标注，例如“文本”、“问题”等等。
  • 步骤—概述大语言模型为生成输出应遵循的处理步骤。
  • 语气—指定大语言模型回答时您期望的语气，如正式、非正式、风趣、热情、冷静、友好等，也可组合使用。
  • 输出格式大语言模型默认以文本形式回答。但你也可以要求以表格、CSV 文件（含列名）、JSON（含属性名）或其他结构化格式呈现。
  • 示例—在大语言模型可能缺乏特定任务（尤其是文本分类和推理任务）足够训练的情况下，可通过示例提供小规模内联训练。即使少量示例也能显著提升回答质量。

总结

  • 提示词通过提供指令和上下文（背景信息，比如领域知识或约束条件）来引导大语言模型（LLM）的运行，清晰且具体的提示词往往能比模糊的提示词产生更优的结果。
  • 不同指令模式适用于不同任务：分类型提示词要求给出类别标签或情感评分；生成型提示词要求创作原创性内容；抽取型提示词则从输入文本中提取特定事实。
  • 当提示词未能产生预期输出时，大语言模型可能缺乏该类型任务的训练示例，此时添加少量示例或提升指令的清晰度，通常可有效解决问题。
  • 少样本学习通过示例展示期望的响应模式：单样本（one-shot）使用一个示例，双样本（two-shot）使用两个示例，依此类推——增加示例数量通常能提升输出一致性，但也会相应增加token开销。
  • 标准提示词通常包含以下六个要素：角色设定（角色）、上下文（背景）、指令（任务）、输入（数据）、风格（语气与格式）和示例（模式）。并非每个任务都需要全部包含这6个部分。
  • 你可以根据需要省略、重新排列或扩展提示词的各个部分，并加入自定义元素，如约束条件、输出格式规范或多步推理指令。
  • 基于API的提示词自动化通过将变量注入提示词模板，并以编程方式执行请求，从而将提示词应用从单次请求扩展至成千上万种变体。例如，只需使用一个模板，结合来自数据库或CSV文件的不同产品属性（如名称、类别、功能），即可批量生成数千条产品描述。
  • 提示模板使用占位符（通常为用花括号包裹的 {variable_name}），这些占位符在运行时会被替换为实际值，从而将提示此的结构与动态内容分离。
  • LangChain 提供了三种提示词模板类，PromptTemplate用于基础的变量替换，支持将运行时动态数据注入预定义的模板结构中。FewShotPromptTemplate会根据输入内容的语义相似性，动态选择最相关的示例，实现少样本学习的自适应提示词构建。而ChatPromptTemplate专门用于结构化多轮对话场景。
  • 思维链（CoT）提示词通过要求大语言模型在给出最终答案前，逐步展示其推理过程来提升输出的可解释性与准确性。通常可加入如“让我们逐步思考”之类的指令，或明确列出所需的推理步骤，引导模型进行结构化思考。
  • 使用XML风格的标签来结构化提示词，明确划分各部分内容：<persona>、<context>、<instruction>、<input>、<examples>。此方法遵循 OpenAI 与 Anthropic 的提示词工程规范，旨在提升提示词的清晰度与可操作性。
  • 使用 OpenAI API 时，消息格式需以list列表形式组织，每条消息包含 role 和 content 两个字段： messages=[{"role": "system", "content": "..."}, {"role": "user", "content": "..."}] 支持的角色包括：system、user 和 assistant。
